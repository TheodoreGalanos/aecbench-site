---
title: Backends
description: A compute backend executes a single trial inside a container or sandbox on a target compute platform.
---

Backends are pluggable via the `ComputeBackend` protocol. The same trial produces the same trial record regardless of which backend runs it.

This page covers the compute layer. For the orchestration layer that dispatches jobs to external services, see [Deployment](/docs/advanced/deployment).

## The backend protocol

Every backend implements the same five-method interface:

```python
class ComputeBackend(Protocol):
    def build_environment(self, task_dir: Path) -> ImageRef: ...
    def launch_trial(self, image_ref: ImageRef, workspace_dir: Path) -> TrialHandle: ...
    def execute_trial(self, handle: TrialHandle, request: BackendExecutionRequest) -> BackendExecutionResult: ...
    def collect_outputs(self, handle: TrialHandle) -> CollectedArtifacts: ...
    def teardown(self, handle: TrialHandle) -> None: ...
```

The orchestrator drives the sequence — build the task's container image, launch a runtime for one trial, execute the agent and verifier inside, collect artifacts, tear down. A `TrialHandle` is an opaque identifier: a Docker container ID, a Modal sandbox reference, whatever the backend needs.

## Available backends

Different backends suit different workflows. The CLI's `--backend` flag selects among them:

| Backend | Where trials run | Suited for |
|---------|------------------|------------|
| **`local`** | In-process on the host | Adapter iteration, fast feedback |
| **`modal`** | Modal serverless containers | Scaling out without infra |
| **`docker`** | Local Docker daemon | Reproducible runs on your machine |
| **`e2b`** | E2B sandboxes | Sandbox-per-trial isolation |
| **`daytona`** | Daytona workspaces | Dev-environment-backed runs |

Maturity varies — `modal` and `local` are the paths most of the library is tested on. The others are supported at the protocol level but may need environment tuning for your workloads.

## Local

In-process execution on your machine. The adapter runs in Python directly; the verifier runs as a subprocess against a local workspace directory. No containerisation.

```yaml
compute:
  backend: local
```

Zero infrastructure setup. Good for adapter development, debugging verifier scripts, and short experiments where you'd rather see stack traces than log artefacts. Throughput is bounded by a single host.

## Modal

Each trial runs in an ephemeral Modal sandbox built from the task's environment image. Task fixtures stage onto a Modal volume; outputs pull back when the trial completes.

```yaml
compute:
  backend: modal
  modal:
    environment: aec-bench-prod
    timeout_sec: 600
```

Modal scales horizontally without any infrastructure work — hundreds of trials in parallel is a flag change, not a project. Cost is per-second, so short trials are cheap and long ones show up on the bill.

## Docker

A local Docker daemon runs each trial in a container built from the task's Dockerfile. Most of the same guarantees as Modal (reproducible environment, isolation) without leaving your machine.

```yaml
compute:
  backend: docker
```

Good middle ground between `local` (no isolation) and `modal` (cloud cost + latency). Throughput bounded by host resources.

## E2B and Daytona

Both provide per-trial sandboxed environments hosted off-machine:

- **E2B** — purpose-built sandboxes for agent workloads, quick to spin up per trial
- **Daytona** — dev-environment-style workspaces, fit where trials need longer-lived state or heavier tooling

Config follows the same pattern:

```yaml
compute:
  backend: e2b        # or daytona
```

Reach for these when Modal doesn't fit — different regions, different sandbox semantics, or existing infrastructure investments.

## Writing a custom backend

Adding a backend means implementing the five-method protocol:

```python
from pathlib import Path
from aec_bench.harness import ComputeBackend, ImageRef, TrialHandle
from aec_bench.harness.backends import BackendExecutionRequest, BackendExecutionResult

class KubernetesBackend:
    def build_environment(self, task_dir: Path) -> ImageRef:
        # push image to registry, return ref
        ...

    def launch_trial(self, image_ref, workspace_dir) -> TrialHandle:
        # create pod, return handle
        ...

    def execute_trial(self, handle, request) -> BackendExecutionResult:
        # exec adapter + verifier in pod, return result
        ...

    def collect_outputs(self, handle):
        # pull logs and reward artifacts
        ...

    def teardown(self, handle):
        # delete pod
        ...
```

Once registered, the new backend is selectable from any experiment config. Trial semantics don't change — same adapter, same verifier, same trial record at the other end.
